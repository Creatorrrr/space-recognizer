# 수정 방안: 기울어짐 / 루프 미폐합 / 포인트클라우드 가림

> 작성: 2026-06-12. 실촬영 영상(office-loop 등) 검증에서 보고된 3가지 문제의
> 원인 분석과 단계적 수정 계획. 실측 근거는 `docs/benchmarks.md` 참고.

## 요약

| 문제 | 근본 원인 | 핵심 수정 | 난이도 |
|---|---|---|---|
| 1. 수평 보행인데 3D가 위/아래로 기울어짐 | 전역 좌표계 = 첫 카메라 자세 (중력 정렬 없음) + pitch drift | 바닥 평면 기반 중력 정렬 (시작 1회 + 저주기 서보) | 중 |
| 2. 시작 지점으로 돌아왔는데 궤적이 안 닫힘 | (a) 궤적 시각화가 과거를 다시 안 그림 (b) 루프 엣지가 1회용이라 부분 보정 잔여분이 수렴 못 함 | (a) 보정된 kf pose로 궤적 재로깅 (b) 루프 엣지 영속화 + 반복 수렴 | 중 |
| 3. 포인트클라우드가 가려서 안 보임 | 지도·미리보기·GS가 한 뷰에 전부 겹침, 토글 없음 | config 토글 + 최근 epoch만 표시 + 블루프린트 분리 | 하 |

권장 작업 순서: **3 → 2 → 1** (3은 반나절·무위험, 2는 사용자가 곧바로 체감,
1은 기하 변경이라 검증 비용이 가장 큼).

## 문제 1 — 수평 보행인데 3D 재구성이 위/아래로 기울어짐

### 원인 분석

1. **전역 좌표계가 첫 카메라 자세로 정의된다** (지배적 원인).
   `vo.py`의 VO는 `T_wc = I`로 시작하고, 전역 좌표계는 이 live 좌표계를
   따른다 (`worldmap.T_global_live`, 첫 윈도가 전역을 정의 — `backend.py`).
   손에 든 폰은 보통 아래로 10~20° 기울어 있으므로, **세계 전체가 그
   pitch만큼 기울어진 채 시작**한다. 수평으로 100걸음 걸으면 세계
   좌표에서는 `100보 × sin(15°) ≈ 26보`만큼 "아래로 내려간" 것으로 보인다.
   증상("크게 기울어짐")과 정확히 일치한다.
2. **VO pitch drift** (부차적). 단안 VO의 회전 오차가 누적되어 기울기가
   시간에 따라 변한다. 루프 클로저가 잡아주지만 재방문 없는 구간은 누적된다.
3. 어디에도 중력/바닥 기준이 없다 — IMU 없음(영상 파일), 바닥 평면 추정
   없음. DA3 depth가 조밀하므로 바닥 평면은 신뢰성 있게 추정 가능한데
   쓰지 않고 있다.

### 수정 방안 (단계적)

#### Phase 1A — 시작 시 1회 중력 정렬 (효과 큼, 위험 낮음)

첫 키프레임의 보정된 depth로 바닥 평면을 찾아, 그 법선이 세계 -Y(위쪽,
RDF 규약)가 되도록 live 좌표계의 초기 자세를 회전시킨다.

- **구현 위치**: 신규 `floor.py` + `main.py`의 K 워밍업 직후.
  1. `estimate_floor(depth, K, cam_pitch_hint=None) -> (normal, d, inlier_frac) | None`
     - 이미지 하단 60% 픽셀을 stride 4로 역투영 → RANSAC 평면 피팅
       (3점 샘플, 거리 임계 = 0.02 × median depth, 200회).
     - 게이트: inlier 비율 ≥ 25%, 법선이 카메라 -Y축과 60° 이내
       (벽을 바닥으로 오인 방지), 평면까지 거리 d > 0 (카메라가 평면 위).
  2. 첫 키프레임 생성 직전에 `R0 = rotation_aligning(normal_world → [0,-1,0])`
     를 계산해 `vo.T_wc` 초기값을 `R0`로 설정 (현재 `np.eye(4)`).
     이후 모든 pose/지도/객체가 자동으로 중력 정렬된 좌표계에 놓인다.
  3. 실패 시(게이트 미달) 현행 동작 유지 + 로그 1줄 — 회귀 없음.
- **config**: `vo.gravity_align: true` (기본 true 가능 — 실패 시 무동작).
- **검증**:
  - 단위: 합성 depth(평면 + 노이즈 + 기울어진 카메라)에서 R0 적용 후
    바닥 법선이 -Y와 1° 이내.
  - e2e: office-loop에서 궤적의 Y 변동폭이 정렬 전 대비 크게 감소
    (수평 보행이므로 이상적으로는 ±수 cm 등가). 톱다운/측면 투영 비교.
- **리스크**: 시작 화면에 바닥이 거의 안 보이는 경우 → 게이트가 막아주고,
  Phase 1B가 나중에 보정한다.

#### Phase 1B — 저주기 자세 서보 (pitch drift 지속 보정)

스케일 서보(`backend.scale_servo`)와 동일한 패턴으로, 백엔드 윈도마다
전역 지도에서 바닥 평면을 추정해 기울기 잔차를 소량씩 되돌린다.

- **구현 위치**: `backend.py` `_run_window()` 끝부분 + `worldmap.py`.
  1. 이번 윈도의 융합 포인트(`points`, 전역 좌표)에서 바닥 평면 추정
     (Phase 1A 함수 재사용; 단 전역 -Y와 30° 이내 법선만 후보 — 이미
     대략 정렬된 상태에서 잔차만 측정).
  2. 기울기 각 θ = angle(normal, -Y). θ > 1°이고 inlier 충분하면
     보정 회전 `R_c` (수평축, θ를 줄이는 방향)를 **윈도당 ≤ 2°로 클램프**해
     `BackendResult.attitude_corr: Sim3`로 전달.
  3. main이 `worldmap`의 `T_global_live` target에 합성 — **현재 카메라
     위치를 고정점으로** 회전시켜야 장면이 휙 돌지 않는다:
     `C = (1, R_c, p_cam - R_c @ p_cam)`.
- **config**: `backend.attitude_servo: true|false` (기본 false → 검증 후 true).
- **검증**: 인위적으로 기울인 시작(Phase 1A 끔) + 긴 영상에서 θ 시계열이
  0으로 수렴하는지. 계단/경사로 영상에서 오동작하지 않는지(게이트 확인).
- **리스크**: 경사 바닥·책상 위주 장면에서 잘못된 평면을 바닥으로 오인 —
  법선 근접 게이트(30°) + 평면 크기 게이트(inlier 수 하한)로 방어.
  스케일 서보와 같은 "frame_scale 상쇄" 문제는 없음 (회전은 live 내부
  상태와 무관하게 T_global_live에서만 처리 가능).

#### Phase 1C (선택) — depth 기반 per-frame 수평 힌트

DA3METRIC 모델은 normal도 줄 수 있는 변형이 있으나 현재 패키지 출력에는
없음. 필요 시 frame별 바닥 추정으로 1B의 주기를 좁힐 수 있으나,
1A+1B로 충분할 것으로 예상 — 보류.

## 문제 2 — 시작 지점으로 돌아왔는데 궤적이 닫히지 않음

### 원인 분석

1. **시각화가 과거 궤적을 다시 그리지 않는다** (체감 원인의 절반 이상).
   `viz.py:77-81`: `_trajectory`는 "그 시점에 기록된 전역 위치"의
   append-only 리스트이고, 매 프레임 전체 폴리라인을 재로깅한다. 루프
   클로저가 과거 키프레임 pose를 보정해도(`kf_global_poses`) **이미
   기록된 궤적 점은 그대로** — 내부적으로 폐합이 진행돼도 화면의 선은
   영원히 안 닫힌다. 지도 voxel은 epoch 보정으로 이동하는데 궤적만
   안 움직이는 비대칭.
2. **루프 엣지가 1회용이다**. `backend.py._close_loops()`는 이번 윈도에서
   수락된 루프만 엣지로 넣고 버린다. 점진 보정(λ<1)으로 잔여분이 남은 채
   카메라가 재방문 지점을 지나가 버리면 **새 루프 엣지가 더 안 생겨
   잔여분이 영영 수렴하지 못한다**. office-loop 실측에서 말미 루프가
   λ=0.07로 부분 적용된 것이 정확히 이 상황 — 93%의 보정이 미적용으로
   남을 수 있다.
3. (배경) 스케일 drift가 크면 외향/귀환 경로의 스케일이 달라 완전 폐합이
   원리적으로 어려움 — 스케일 서보가 1차 완화(2026-06-12), 잔여는 루프의
   스케일 성분이 처리.

### 수정 방안 (단계적)

#### Phase 2A — 보정된 키프레임 pose로 궤적 재로깅 (반나절, 체감 최대)

- **구현 위치**: `viz.py` + `main.py` 드레인.
  1. `viz.log_trajectory_correction(kf_poses: dict[int, np.4x4], kf_ts: dict[int, float])`
     신규: 백엔드 결과가 올 때마다 **키프레임 전역 pose들로 궤적 폴리라인을
     통째로 다시 로깅**한다 (현재 프레임까지는 마지막 kf pose→현재 pose
     보간 1구간 추가).
  2. `BackendResult.kf_global_poses`는 이미 전달되고 있음 — kf ts만
     함께 보내면 됨 (worker가 `_epoch_kfs`처럼 `kf_ts: dict[int,float]`
     유지, result에 포함).
  3. 기존 per-frame `_trajectory` 누적은 "최근 구간"(마지막 kf 이후)
     전용으로 축소 — 백엔드 결과 사이의 부드러운 선 유지.
- **효과**: 루프가 발화하는 순간 화면의 궤적이 닫히는 것이 보인다.
  내부 보정과 화면이 일치하게 되는 것이 본질.
- **검증**: 부메랑 영상(정답: 닫힘)에서 보정 후 궤적 시작/끝 간 화면
  거리 ≈ 0. office-loop에서 재방문 구간 겹침 확인.
- **리스크**: 낮음 — 시각화 전용, 기하 무변경.

#### Phase 2B — 루프 엣지 영속화 + 반복 수렴 (1~2일, 실폐합의 핵심)

- **구현 위치**: `backend.py` `_Worker`.
  1. `self._loop_edges: list[(old_id, new_id, T_ab, weight, age)]`에 수락된
     루프를 **보관**한다 (상한 예: 최근 50개, 같은 쌍 중복은 inlier 큰
     것으로 교체).
  2. `_close_loops()`가 **새 루프가 없어도** 보관된 엣지의 잔차가 크면
     pose graph를 재실행한다: 잔차 = `log(inv(Z) ∘ inv(P_i) ∘ P_j)`의
     norm 최댓값. `> 0.05`(병진+스케일 합성 norm) 동안 매 윈도 반복 →
     λ 부분 적용이 윈도를 거치며 자동 수렴. 잔차가 작아지면 휴면.
  3. 보정 후 `T_ab` 측정치는 불변(측정), 노드 pose만 갱신 — 수렴 보장
     (gauge 고정 + LM).
  4. 가드: 보관된 엣지의 한쪽 kf가 pose graph 노드 솎아냄에서 빠지지
     않게 `must` 집합에 포함 (이미 있는 메커니즘 확장).
- **config**: `loop.persist_edges: true` (기본 true — 동작은 보수적:
  잔차 없으면 아무 것도 안 함).
- **검증**: 부메랑에서 λ<1 보정 후 2~4윈도 내 잔차 < 0.05 수렴.
  office-loop에서 말미 대형 루프(λ=0.07)가 영상 끝 드레인 동안 추가
  윈도들로 수렴해 폐합 잔차 감소.
- **리스크**: 오탐 루프가 영속화되면 지도를 계속 잡아당김 — 수락 게이트가
  이미 보수적(인라이어 15+합의율 45%)이고, 영속 엣지에 **age 감쇠 가중**
  (오래된 엣지 weight × 0.9/윈도, 하한 0.3)으로 완화.

#### Phase 2C — 고신뢰 루프의 스냅 + 드레인 연장 (반나절)

  1. 인라이어가 압도적인 루프(예: ≥ 150 — 부메랑/office 말미에서 실측
     200~1000)는 λ 클램프를 완화 (`_MAX_STEP_T × 3`) — 명백한 재방문은
     빠르게 닫는다. 오탐 가능성이 사실상 0인 증거 수준에서만.
  2. `main.py` 영상 종료 드레인: 루프 잔차가 수렴 중이면 드레인 시간을
     연장 (현재 8초 고정 → "백엔드가 잔차 수렴 신호를 보낼 때까지 최대
     30초"). 영상이 재방문 직후 끝나는 경우(흔함)의 미수렴 방지.
- **검증**: office-loop 말미 루프가 종료 전에 완전 수렴하는지.

#### 우선순위 메모

2A 없이 2B만 하면 "내부는 맞는데 화면이 틀린" 상태가 지속되므로 **2A를
먼저**. 2B는 2C의 전제.

## 문제 3 — 포인트클라우드가 많아서 3D 재구성이 가려짐

### 원인 분석

한 3D 뷰(`world/`)에 다음이 전부 겹쳐 그려진다:

| 엔티티 | 점 수 | 비고 |
|---|---|---|
| `world/points` (voxel 지도) | 최대 200만 | radii 0.006 |
| `world/live_preview` (최신 키프레임 미리보기) | ~5.7만/kf | radii 0.008 — 카메라 앞을 항상 덮음 |
| `world/gaussians` (GS 레이어) | ~20만 | radii 0.005 |
| 객체 노드/그래프/궤적/카메라 | 소량 | 정작 잘 안 보이게 됨 |

Rerun UI에서 수동으로 눈 아이콘 토글은 가능하지만 기본값이 전부 켜짐이고,
"최근 것만 보기" 같은 필터는 없다. 단 `GlobalMap`이 voxel별 마지막 갱신
epoch(`_epoch`)을 이미 추적하므로(Tier 4) **최근-만 필터의 기반은 이미 있다**.

### 수정 방안 (단계적)

#### Phase 3A — config 토글 + 블루프린트 기본 가시성 (2~3시간)

- `VizCfg`에 추가:
  ```yaml
  viz:
    show_map_points: true     # world/points
    show_live_preview: false  # world/live_preview (기본 끔 권장 — 가림 주범)
    show_gaussians: true      # world/gaussians
  ```
- `viz.py`: 끈 엔티티는 로깅 자체를 생략 (대역폭·메모리도 절약).
  블루프린트에서 `rrb` 가시성 오버라이드(`EntityBehavior` 또는
  `Spatial3DView(contents=...)` 제외 방식 — rerun-sdk 0.33 API로 구현 시
  확인)로 "로깅은 하되 기본 숨김"도 가능 — 사용자가 뷰어에서 켤 수 있게
  하려면 이 방식이 낫다. **권장: live_preview는 기본 숨김 오버라이드,
  나머지는 로깅 토글.**
- **검증**: 켜고 끈 상태의 뷰어 확인. 끔 상태에서 성능 회귀 없음(당연히
  빨라짐).

#### Phase 3B — "최근 N윈도만" 지도 표시 모드 (반나절)

- `VizCfg.map_recent_epochs: int = 0` (0 = 전체, N>0 = 최근 N epoch만).
- `GlobalMap`에 `recent_points(n_epochs, current_epoch)` 추가:
  `mask = self._epoch > current_epoch - n_epochs` → points/colors 필터.
  (epoch 배열은 이미 있음 — 필터 한 줄.)
- `main.py` 드레인에서 `viz.log_global_map(...)`에 필터 적용해 로깅.
  전체 지도는 `world/points_full`로 기본-숨김 로깅해 뷰어에서 선택
  가능하게 (선택).
- **효과**: "최근 프레임에 대한 포인트클라우드만" 요구사항을 epoch
  단위(5초 윈도)로 충족. 카메라가 보고 있는 곳 위주로만 표시된다.
- **검증**: office-loop에서 N=3으로 최근 ~15초 구간만 표시되는지.

#### Phase 3C (선택) — 카메라 반경 필터 / 표시 서브샘플

- `viz.map_display_radius` (현재 카메라 기준 거리 필터),
  `viz.map_display_subsample` (표시용 추가 서브샘플) — 3A/3B로 부족할 때.
- GS 레이어가 활성일 때는 `world/points`를 기본 숨기고 GS 중심점만
  보여주는 프리셋도 고려 (GS가 더 정돈된 시각화).
